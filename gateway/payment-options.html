Payment And Refund Options

AgentCart should make payments easy for a WooCommerce merchant without pretending that settlement, refunds, FX, chargebacks, and consumer rights disappear. The right production shape is multi-rail: Tempo for stablecoin-capable agents, Stripe/card for normal users and EUR merchants.

Dashboard Presentation Onboarding Field Map Quote Tournament Architecture
Normal buyer setup
  1. Use an agent that can call the ShopBridge catalog, quote, approval, checkout, and order-status APIs.
  2. Store delivery address and household policy in the agent or buyer profile.
  3. Provide a payment credential source: funded MPP wallet, Link/Stripe SPT flow, or another supported MPP method.
  4. Approve the final quote before checkout.
WooCommerce merchant setup
  1. Install WooCommerce and configure products, tax, stock, shipping, terms, and returns as usual.
  2. Install AgentCart ShopBridge and publish the machine-readable manifest.
  3. Choose settlement methods: Tempo stablecoin, Stripe/card, or both.
  4. Configure verifier, recipient/profile, refund handling, and order-status/tracking plugins.
Hackathon boundary

The demo proves opt-in discovery, quote competition, human approval, Woo order creation, pathUSD testnet proof, delivery estimate, Vikunja task, and audit. It does not claim real EUR settlement or production refund automation.

Refund Handling

What WooCommerce already has

WooCommerce already represents refunds in the order admin. The screenshot with the Refund button is useful for the demo because it shows we are creating a real Woo order surface, not a fake standalone receipt.

What AgentCart must add for production

A Woo refund must call the same payment verifier/rail that accepted the payment. Recording a Woo refund is not enough if the money moved through Tempo, Stripe, or another rail.

Refund caseProduction behaviorAgentCart fields to store
Stripe/card Use Stripe refund API or Dashboard. Stripe keeps the payment/refund/reporting/dispute record. stripe_payment_intent, refund_id, amount, currency, status, reason.
Tempo stablecoin Refund to the source wallet or run a rail-specific refund transfer. The verifier should bind the refund to the original receipt and quote/order. transaction_reference, from_wallet, refund_transaction_reference, asset, network.
Demo mode Record refund intent only. Do not claim funds moved. state: demo_refund_recorded, real_refund: false, reason, audit_event_id.

Suggested Product Decision

Include in local demo
  • Show Stripe/card and refunds as production-grade options in the merchant onboarding story.
  • Expose refund_policy and payment_requirements.protocols[] in the merchant manifest/quote.
  • Show the WooCommerce Refund button and explain that real rail refunds are future verifier work.
Do not overbuild before presentation
  • Do not implement live Stripe/card unless credentials and access are ready.
  • Do not claim Woo refunds move pathUSD or EUR unless the rail refund actually executes.
  • Do not make the buyer create a shop account in the browser; collect address and consent through the agent profile/API.
