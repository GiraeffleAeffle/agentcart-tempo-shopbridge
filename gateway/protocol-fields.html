AgentCart Protocol Field Map

MPP is the payment boundary. AgentCart ShopBridge is the commerce and household-safety profile around it. This page shows which fields are official MPP concepts, which are required for merchant checkout, and which are optional local integrations.

1. Official MPP Payment Layer

Discovery

x-payment-info.offers[] with amount, currency, intent, and method. Runtime 402 remains authoritative.

Challenge

402 Payment Required plus WWW-Authenticate: Payment. Carries id, realm, method, intent, encoded request, expiry, and method-specific fields.

Credential

Client retries with Authorization: Payment. Payload is method-specific, for example Tempo transaction/hash/proof or Stripe SPT.

Receipt

Server returns Payment-Receipt after verification. It may include method, status, reference, timestamp, and delivery confirmation metadata.

2. ShopBridge Commerce Profile

Merchant Manifest

merchant_id, domain, manifest_url, manifest_hash, supported protocols, ship-to countries, terms, returns, and merchant-of-record.

Product

product_id, sku, title, description, category, brand, unit_size, images, price hint, stock, availability, delivery countries.

Quote

Final quote with items, subtotal, shipping, VAT lines, total, currency, stock hold, delivery window, quote hash, expiry, and payment requirements.

Order

Merchant order id, order URL, status URL, status token, fulfillment state, delivery estimate, tracking placeholders, and receipt references.

3. Household Safety Profile

Policy

Budget ceilings, approval thresholds, allowed/blocked categories, allowed countries, merchant allowlist/blocklist, and max quantity.

Approval

Consent request, approval id, one-time decision token, expiry, decision API, approver, channel, and decision reason.

Audit

Intent, sensor/task context, selected quote, rejected candidates, policy result, approval, payment receipt, order, and delivery state.

Replay Protection

Quote hash, idempotency key, challenge id, request digest, expiry, receipt reference, and merchant order correlation.

4. Optional Local Interfaces

Home Assistant

Sensor trigger, phone/watch notification, calendar write, and energy telemetry. Optional adapter, not required by ShopBridge.

OpenClaw / LLM

Intent parsing and tool use. The API is deterministic, so another agent can use the same endpoints.

Vikunja

Task lookup, task update, and due-date creation. Optional household aftercare.

Telegram / Signal / Email

Alternative approval delivery channels if they can render the consent request and call the signed decision API.

Field Boundary

QuestionAnswerWhy it matters
Do we need to extend core MPP for products? No. MPP is intentionally payment-method agnostic. Products, VAT, delivery, approval, and orders should stay in a commerce profile around MPP.
When would we extend MPP? Only for a new payment rail, new intent, escrow/refund primitive, or quote-bound FX settlement requirement. Those are payment semantics. Catalog and household policy are application semantics.
Can the WooCommerce plugin support Stripe/card settlement? Yes, by advertising another MPP payment method and verifying a Stripe/card credential before creating the Woo order. Stripe/card gives EUR merchants a production path for payouts, reporting, refunds, disputes, and cardholder protections.
Should refunds be in the profile? Yes, as a production requirement and quote/order capability field. WooCommerce can record refunds, but AgentCart must bind real refunds to the original payment rail before claiming funds moved.
What is the pathUSD problem? Tempo testnet uses pathUSD, while the shop quote is EUR. The demo must say this clearly. Production needs merchant acceptance of USD stablecoin, quote-bound FX, Stripe/card settlement, or an EUR-compatible MPP rail.

Claim To Make

AgentCart does not replace MPP. It defines the missing opt-in merchant and household safety layer that lets an agent compare products, get final quotes, ask for consent, pay with an MPP-compatible method, and leave a clear order/audit trail.
