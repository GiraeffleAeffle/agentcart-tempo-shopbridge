AgentCart

A household-safe bridge between personal agents and opt-in merchants.

The problem is not only "can an agent pay?" MPP answers that. The missing layer is how a normal shop exposes enough product, quote, policy, approval, order, delivery, and audit information for a household agent to buy safely.
Gap

Commerce systems

Household agents

Protocol Boundary

MPP handles payment. AgentCart handles commerce context.

Official MPP fields

402, WWW-Authenticate: Payment, Authorization: Payment, Payment-Receipt, intent, method, amount, currency, challenge expiry, request binding, and payment-method payloads.

AgentCart ShopBridge fields

Product, final quote, VAT, shipping, delivery country, merchant-of-record, policy result, human approval, Woo order, delivery status, Vikunja task, and audit log.

Conclusion: no core MPP extension is needed for product discovery. New payment rails, FX, escrow, or refund semantics would be MPP method/intent work.

ShopBridge Plugin

A normal WooCommerce shop can opt in without scraping.

1. Manifest/.well-known/agentcart.json advertises capability and endpoints.
2. Catalog/catalog exposes product, price hint, stock, images, category, and delivery countries.
3. Quote/quote returns final VAT, shipping, total, delivery window, merchant of record, and quote hash.
4. Paid order/orders creates a Woo order only after quote-bound payment verification.
5. Status/orders/{id}/status returns Woo status, delivery estimate, and tracking metadata when present.

WooCommerce remains the source of truth for tax, stock, shipping, fulfillment, refunds, support, and customer rights.

Merchant Onboarding

The store owner path should fit into WordPress admin.

Install

Install WooCommerce, add products as usual, then activate AgentCart ShopBridge.

Configure

Set support email, shipping countries, Tempo recipient, payment verifier URL, and gateway token in WooCommerce -> AgentCart.

Publish

Publish the manifest URL and optionally anchor merchant identity plus manifest hash in a registry.

Test

Run sandbox catalog, quote, approval, MPP payment proof, Woo order, and status retrieval.

Operate

Fulfill orders in WooCommerce, add tracking through normal shipment plugins, handle refunds in the existing store flow.

Target first

Local tea and coffee shops, refill stores, farm shops, club merch, and niche D2C stores already running WordPress.

Discovery And Competition

Agents need fair quote discovery, not ad-driven browsing.

Registry as identity anchor

Keep public records small: domain, merchant id, manifest URL, manifest hash, payment network, recipient, and version. Keep catalog, stock, customer data, and prices offchain.

Private quote tournament

The household agent sends exact terms to matching merchants. Merchants return final quotes. The policy engine chooses by total price, delivery country, category, trust, and consent rules.

This has intent-market similarities, so the MVP avoids public bidding wars. No hidden scraping, no pay-to-rank ads, and no order without quote-bound approval.

Safety

The demo is intentionally constrained.

No scraping

No browser automation against real shops. The merchant is opt-in and fake for the hackathon.

No silent spend

Household policy requires approval. Budgets and categories are enforced before checkout.

No fake settlement claims

WooCommerce quotes in EUR. The Tempo MPP proof is pathUSD testnet value transfer, not EUR settlement of the tea order.

Payment Rails

Small shops need a choice: wallet-native or Stripe-native.

Tempo MPP

Best for agent-native stablecoin settlement. Buyer needs an MPP-capable funded wallet; merchant needs recipient address, verifier, accounting, and refund handling.

Stripe/card MPP

Best for normal users and EUR shops. Buyer uses a Shared Payment Token or Link/card flow; merchant uses Stripe profile, reporting, payouts, refunds, and disputes.

Refunds

WooCommerce can show the refund workflow, but production must execute the refund through the original rail before claiming funds moved.

Hackathon claim: we support the interface shape and Woo order surface. Real Stripe/card and rail-bound refunds require merchant credentials and verifier implementation.
